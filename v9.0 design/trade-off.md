Trade-off
1 设计决策矩阵
决策点	选择	权衡
签名算法	HMAC-SHA256	安全性高 ✅ / 计算开销稍大（~0.1ms）
防重放存储	Redis	分布式共享 ✅ / 增加网络开销 + Redis 依赖
Nonce 长度	16 字节 (32 hex)	碰撞概率极低 ✅ / 传输开销略增
时间窗口	60 秒	容忍网络延迟 ✅ / 重放窗口较大
原子操作	SETNX (Redis)	天然原子性 ✅ / 依赖 Redis 可用性
中间件顺序	Limiter → Sign → Auth	限流优先可防攻击 ✅ / 签名校验放在认证前（更早拒绝）
2 安全性 vs 性能
维度	损失	收益
延迟	+1-2ms (签名计算 + Redis)	防重放 + 防篡改
QPS	-10~15%	请求完整性保证
复杂度	+1 中间件 + Redis 依赖	分布式场景一致
3 分布式 vs 单机
方案	优点	缺点
单机内存 (sync.Map)	无网络开销	多实例无法共享
Redis 共享	全局一致	增加依赖 + 网络延迟
JWT 自包含	无状态	无法主动撤销，窗口内可重放
4 成本分析
text
单请求额外开销：
├── 签名计算：~0.1ms (CPU)
├── Redis SETNX：~0.5ms (网络)
├── 时间戳校验：~0.01ms
└── 总计：~0.6ms

对比无签名接口 QPS 下降：约 10-15%
5 错误码设计
错误码	含义	客户端处理
4091	缺少签名头	检查请求构造
4093	Nonce 非法	重新生成 16-32 字节随机数
4094	时间戳超时	同步时间，重新请求
4096	签名无效	检查 secret 和 payload
4098	Nonce 已使用	生成新 nonce 重试
6 降级策略
text
Redis 不可用时：
├── FailOpen = true  → 放行所有请求（降级）
├── FailOpen = false → 返回 500（快速失败）
└── 建议：生产环境 FailOpen = false，配合 Redis 哨兵
7 与其他中间件的交互
text
请求进入 → 
  1. Panic 恢复：捕获签名计算时的异常
  2. 链路追踪：trace_id 贯穿全程
  3. 限流：优先于签名（防止攻击者消耗 Redis）
  4. 签名验证：拦截无效请求
  5. 认证：最后验证（签名可防伪造，认证确定身份）
